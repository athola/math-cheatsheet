## 1. Foundations of Equational Logic

### 1.1 Syntax and Semantics

**Equational Logic** deals with universally quantified equations over algebraic signatures.

**Definition (Equation):** An equation is a pair of terms (t₁, t₂) written as t₁ ≐ t₂, where t₁, t₂ are terms built from:
- Variables: Var = {x, y, z, ...}
- Function symbols: F = {f, g, h, ...} with arities
- Constructors: If f ∈ F has arity n and t₁, ..., tₙ are terms, then f(t₁, ..., tₙ) is a term

**Example Equations:**
- Associativity: (x · y) · z ≐ x · (y · z)
- Commutativity: x · y ≐ y · x
- Identity: e · x ≐ x

**Semantics:** An equation e = (t₁ ≐ t₂) holds in an algebra A (written A ⊧ e) if for every assignment σ: Var → A, the interpretation of t₁ under σ equals the interpretation of t₂ under σ.

### 1.2 Equational Implication

**Definition (Implication):** A set of equations E₁ implies equation e₂ (written E₁ ⊧ e₂) if every algebra satisfying all equations in E₁ also satisfies e₂.

**Key Properties:**
- **Reflexivity**: E ⊧ E (trivial)
- **Transitivity**: If E₁ ⊧ E₂ and E₂ ⊧ E₃, then E₁ ⊧ E₃
- **Monotonicity**: If E₁ ⊧ e and E₁ ⊆ E₁', then E₁' ⊧ e
- **Compactness**: If E ⊧ e, then some finite E' ⊆ E satisfies E' ⊧ e

### 1.3 Birkhoff's HSP Theorem

**Theorem (Birkhoff's HSP):** A class of algebras K is definable by equations (is a variety) iff K is closed under:
- **H**omomorphic images
- **S**ubalgebras
- **P**roducts

**Corollary:** E₁ ⊧ e₂ iff the variety V(E₁) generated by E₁ is contained in V({e₂}).

**Significance:** This provides the foundational connection between syntactic implication and semantic classes of algebras.

### 1.4 Decidability Landscape

**Undecidable Cases:**
- General equational implication (Redko, 1967)
- Word problem for finite semigroups (Markov, Post)
- Confluence of term rewrite systems

**Decidable Cases:**
- Ground equations (no variables)
- Monadic equations (arity ≤ 1)
- Shallow theories (nesting depth bounded)
- Commutative theories with restrictions
- Theories with finite models

## 2. Automated Theorem Proving Systems

### 2.1 Architecture of ATP Systems

**Core Components:**

1. **Parsing and Preprocessing**
   - Input: Equations in various syntaxes
   - Output: Internal representation (terms, clauses)
   - Tasks: Skolemization, CNF conversion, flattening

2. **Reasoning Engine**
   - Inference rules application
   - Search strategy
   - Heuristic guidance

3. **Term Indexing**
   - Subterm indexing
   - Path indexing
   - Discrimination trees

4. **Simplification**
   - Demodulation (rewriting with given equations)
   - Subsumption
   - Tautology deletion

### 2.2 Major ATP Families

#### 2.2.1 Resolution-Based Provers

**Key Systems:**
- **Vampire**: World-leading ATP, equational reasoning via superposition
- **E**: Prover with excellent support for equational reasoning
- **Prover9**: Based on resolution and paramodulation

**Superposition Calculus:**
```
    Maximal literal selection
    + Ordered resolution
    + Ordered paramodulation
    + Simplification rules
```

**Advantages:**
- Strong theoretical foundation
- Proven effectiveness
- Good for first-order logic with equality

**Limitations:**
- Can explode on equational problems
- Requires careful ordering
- Not interactive

#### 2.2.2 Term Rewriting Provers

**Key Systems:**
- **Waldmeister**: Specialized for unit equational problems
- **SPSSat**: Combines SAT with term rewriting
- **Muscadet**: Rewriting-based theorem prover

**Approach:**
1. Convert equations to rewrite rules
2. Orient equations using reduction order
3. Complete using critical pair computation
4. Normalize terms using completed system

**Advantages:**
- Efficient for pure equational logic
- Produces canonical forms
- Decision procedure for decidable theories

**Limitations:**
- Requires termination of rewrite systems
- Not complete for general first-order logic
- Ordering can be difficult

#### 2.2.3 SMT-Based Provers

**Key Systems:**
- **Z3**: SMT solver with quantifier support
- **CVC5**: Successor to CVC4, strong on theories
- **Yices2**: Efficient SMT solver
- **Boolector**: Bit-vector and array theories

**Approach:**
- Ground universal equations (E-matching)
- Use congruence closure for equality
- Combine theory solvers (Nelson-Oppen)
- Model-based quantifier instantiation

**Advantages:**
- Fast for ground problems
- Good combination of theories
- Incremental solving

**Limitations:**
- Quantifier handling is weak
- Grounding can lose information
- Not designed for deep equational reasoning

### 2.3 ATP for Equational Implication

**Problem Transformation:**

To check E₁ ⊧ e₂:
1. Negate: Does there exist a model of E₁ that falsifies e₂?
2. Translate to clause form:
   - E₁ becomes positive unit clauses
   - ¬e₂ becomes t₁ ≠ t₂
3. Search for contradiction using refutation

**Proof Strategies:**

1. **Direct Proof**: Derive e₂ from E₁ using rewrite rules
2. **Refutation**: Assume E₁ ∧ ¬e₂, derive contradiction
3. **Model Construction**: Find countermodel to E₁ ⊧ e₂

**Success Rates:**
- Simple implications: 80-95% automated
- Medium complexity: 50-70% automated
- Hard problems: 20-40% automated (requires guidance)

## 3. Term Rewriting Systems

### 3.1 Foundations

**Definition:** A term rewrite system (TRS) is a set of rewrite rules ℓ → r where:
- ℓ, r are terms
- Var(r) ⊆ Var(ℓ) (variables on right must appear on left)
- ℓ is not a variable (prevents infinite rewriting)

**Example:**
```
x · e → x           (left identity)
x · (y · z) → (x · y) · z   (associativity)
x · y → y · x       (commutativity)
```

**Key Properties:**

1. **Termination**: No infinite rewrite sequences
2. **Confluence**: If t →* u and t →* v, then ∃w: u →* w and v →* w
3. **Joinability**: t₁ and t₂ can be rewritten to common term
4. **Normal Forms**: Irreducible terms

**Church-Rosser Theorem:** A terminating TRS is confluent iff every term has a unique normal form.

### 3.2 Reduction Orders

**Purpose:** Orient equations ℓ ≐ r as rewrite rules ℓ → r

**Key Properties Required:**
- Well-foundedness (no infinite descending chains)
- Closure under substitutions
- Closure under contexts
- Monotonicity

**Major Orderings:**

1. **Simplification Orders**
   - **Recursive Path Ordering (RPO)**: Based on precedence and subterm relations
   - **Knuth-Bendix Ordering (KBO)**: Weights symbols, uses lexicographic comparison
   - **Polynomial Orderings**: Interpret terms as polynomials

2. **Semantic Orders**
   - **Interpretation Orders**: Use algebraic interpretations
   - **Matrix Interpretations**: Linear algebra over natural numbers

3. **Higher-Order Orders**
   - **Higher-Order Recursive Path Ordering (HORPO)**: For lambda calculus
   - **Dependency Pairs**: For termination proofs

### 3.3 Critical Pairs

**Definition:** Given rules ℓ₁ → r₁ and ℓ₂ → r₂, a critical pair arises when ℓ₁ and ℓ₂ overlap.

**Computation:**
1. Find non-variable subterm of ℓ₁ that unifies with ℓ₂
2. Compute most general unifier σ
3. Rewrite ℓ₁σ using both rules: r₁σ and ℓ₁[ℓ₂]σ → ℓ₁[r₂]σ
4. Critical pair: (r₁σ, ℓ₁[r₂]σ)

**Example:**
```
Rules: x · e → x  and  x · (y · z) → (x · y) · z

Overlap: (x · y) · e
Using rule 1: (x · y) · e → x · y
Using rule 2: (x · y) · e matches x · (y · z) with z=e
            → x · (y · e)

Critical pair: (x · y, x · (y · e))
```

**Critical Pair Lemma:** If all critical pairs are joinable, the system is locally confluent.

### 3.4 Completion Procedures

**Basic Knuth-Bendix Algorithm:**
```
Input: Equations E, reduction order >
Output: Complete, convergent TRS R

1. R := ∅
2. While E ≠ ∅:
     a. Pick s ≐ t from E
     b. If s > t:
          Add s → t to R
          Compute critical pairs with new rule
          Add unjoinable pairs to E
        Else if t > s:
          Add t → s to R
          Compute critical pairs
          Add unjoinable pairs to E
        Else:
          Report failure (cannot orient)
3. Return R
```

**Challenges:**
- **Failure to orient**: Not all equations can be oriented
- **Non-termination**: Process may not halt
- **Rule explosion**: Can generate many rules

### 3.5 Advanced Rewriting

#### 3.5.1 Conditional Rewriting

**Rules with Conditions:**
```
x > y → x  IF x > y  (not well-founded!)
x + 0 → x
```

**Applications:**
- Algebraic specifications
- Program transformation
- Constraint solving

#### 3.5.2 Higher-Order Rewriting

**Extension to lambda calculus:**
- Binders and variables
- Beta-reduction as rewrite rules
- Alpha-equivalence handling

**Systems:**
- **HOL**: Higher-order logic
- **Isabelle**: Higher-order rewriting tactics
- **Lambda Prover**: Higher-order logic programming

#### 3.5.3 Constrained Rewriting

**Rules with Constraints:**
```
x + y → y + x  IF x,y ∈ CommSort
```

**Benefits:**
- More expressive than pure rewriting
- Can handle partial functions
- Integrates with constraint solvers

## 4. Completion Procedures

### 4.1 Classical Knuth-Bendix

**Original Algorithm (1970):**

1. **Orient equations** using reduction order
2. **Compute critical pairs** between all rule pairs
3. **Add derived rules** for unjoinable pairs
4. **Repeat until** no new rules generated

**Termination Issues:**
- May not terminate (infinite rule sets)
- Depends heavily on ordering choice
- Sensitive to equation processing order

**Practical Improvements:**
- **Fairness strategies**: Process all critical pairs eventually
- **Rule simplification**: Delete redundant rules
- **Critical pair criteria**: Ignore trivial pairs

### 4.2 Ordered Completion

**Improvement (Bachmair, Dershowitz, Hsiang 1986):**

**Key Idea:** Process equations without requiring total ordering upfront.

**Algorithm:**
```
1. E := {equations}, R := {rules}
2. While E ≠ ∅ or there are new critical pairs:
     a. Pick s ≐ t from E or critical pairs
     b. If s > t: R := R ∪ {s → t}
     c. Else if t > s: R := R ∪ {t → s}
     d. Else: E := E ∪ {s ≐ t}
     e. Simplify E and R using R
     f. Compute critical pairs for new rule
3. Return (E, R)
```

**Advantages:**
- More flexible than basic K-B
- Handles non-orientable equations
- Better theoretical properties

### 4.3 Basic Completion

**Objective:** Minimize dependency on ordering.

**Approach:**
- Maintain equations in non-oriented form
- Compute critical pairs flexibly
- Use ordering only when necessary

**Variants:**
- **Basic paramodulation**: For first-order theorem proving
- **Basic superposition**: Optimization of paramodulation
- **Basic narrowing**: For equational unification

### 4.4 Improved Completion

**Huet's Completion (1981):**

**Improvements:**
1. **Lazy critical pair computation**: Compute on-demand
2. **Rewrite rules**: Simplify equations and rules
3. **Rule overlap criteria**: Efficient critical pair detection

**Algorithm Complexity:**
- Time: O(n³) in worst case (n = number of rules)
- Space: O(n²) for storing critical pairs
- Practical: Often much faster due to heuristics

### 4.5 Completion Variants

#### 4.5.1 AC Completion

**Problem:** Associative-commutative theories break standard completion.

**Solution:** Treat AC as built-in:
- Equivalence classes under AC
- Normalization modulo AC
- AC-unification for critical pairs

**Applications:**
- Commutative operators (addition, multiplication)
- Set theory operations
- Multiset reasoning

#### 4.5.2 Conditional Completion

**Challenge:** Conditional equations complicate completion.

**Approaches:**
1. **Simplification**: Use conditions to prune computations
2. **Constraint solving**: Integrate with constraint solvers
3. **Transformation**: Convert to unconditional form

#### 4.5.3 Higher-Order Completion

**Extension:** Support lambda calculus and higher-order syntax.

**Key Issues:**
- Alpha-conversion handling
- Higher-order unification
- Beta-reduction integration

**Systems:**
- **Andromeda**: Higher-order proof assistant
- **Hol-light**: Higher-order logic prover

### 4.6 Completion in Practice

**Success Factors:**

1. **Good Reduction Order**
   - KBO for algebraic theories
   - RPO for recursive structures
   - Polynomial orders for termination

2. **Heuristics**
   - Process equations by size
   - Prioritize frequently used rules
   - Delete redundant rules

3. **Implementation**
   - Efficient term indexing
   - Memoization of critical pairs
   - Parallel computation

**Failure Cases:**

1. **Non-orientable equations**: Cannot find suitable ordering
2. **Non-termination**: Infinite rule generation
3. **Combinatorial explosion**: Too many critical pairs

**Modern ATP Integration:**
- Superposition calculus generalizes completion
- SAT-based approaches for ground completion
- SMT integration for theory reasoning

## 5. Decision Procedures

### 5.1 Congruence Closure

**Problem:** Decide validity of ground equational literals (uninterpreted functions).

**Applications:**
- Equality reasoning in SMT solvers
- Type checking
- Compiler optimization

#### 5.1.1 Basic Algorithm

**Input:** Set of equalities and disequalities between ground terms

**Output:** SAT/UNSAT

**Approach:**
1. Build equivalence classes using union-find
2. Process equalities: union terms
3. Check disequalities: fail if equivalent terms are disequal

**Complexity:**
- Time: O(n α(n)) where α is inverse Ackermann
- Space: O(n) for n terms

#### 5.1.2 Decision Procedure with Explanation

**Extension:** Produce proofs for unsatisfiability.

**Algorithm (Nelson-Oppen style):**
1. Maintain congruence closure with justifications
2. Track which equalities imply which
3. Extract proof on conflict

**Applications:**
- Proof-critical systems
- Interpolation
- Model checking

#### 5.1.3 Congruence Closure Modulo Theories

**Combination:**
- Uninterpreted functions + theories
- Example: Arrays, linear arithmetic
- Use Nelson-Oppen combination

### 5.2 Nelson-Oppen Combination

**Problem:** Combine decision procedures for disjoint theories.

**Theories:** T₁, T₂ with disjoint signatures

**Approach:**
1. Run each theory solver independently
2. Exchange equalities on shared variables
3. Iterate until fixpoint

**Requirements:**
- **Convexity**: For each theory, if t₁ = s₁ ∧ ... ∧ tₙ = sₙ ⊧ x = y, then some subset implies x = y
- **Stably infinite**: Models can be extended with new elements

**Complexity:**
- Polynomial in convex theories
- Exponential in non-convex theories

**Applications:**
- SMT solvers (Z3, CVC5)
- Verification conditions
- Hardware verification

### 5.3 Theory-Specific Decision Procedures

#### 5.3.1 Linear Arithmetic

**Linear Integer Arithmetic (LIA):**
- Presburger arithmetic (decidable)
- Complexity: O(n⁴) using Cooper's algorithm
- Practical: O(n²) with Omega test

**Linear Real Arithmetic (LRA):**
- Fourier-Motzkin elimination
- Complexity: O(2ⁿ) worst case
- Practical: O(n³) using Simplex

**Applications:**
- Timing analysis
- Resource reasoning
- Array bounds checking

#### 5.3.2 Arrays

**Theory of Arrays:**
- Axioms: extensionality, read-over-write
- Decision procedure using array property elimination
- Complexity: NP-complete

**Applications:**
- Program verification
- Memory safety
- Data structure reasoning

#### 5.3.3 Bit-vectors

**Theory:**
- Fixed-width integers
- Operations: &, |, ^, <<, >>
- Decision: SAT-based translation

**Complexity:**
- NP-complete (reduction to SAT)
- Practical: Bit-blasting + SAT solver

**Applications:**
- Hardware verification
- Cryptographic protocols
- Compiler optimization

#### 5.3.4 Algebraic Data Types

**Theory:**
- Constructors, selectors, testers
- Axioms: injectivity, disjointness
- Decision: Congruence closure + case analysis

**Applications:**
- Functional program verification
- Type system reasoning
- Proof assistants

### 5.4 Quantifier Elimination

**Problem:** Remove quantifiers from formulas.

**Techniques:**

1. **Cylindrical Algebraic Decomposition (CAD)**
   - Theory: Real closed fields
   - Complexity: Doubly exponential
   - Practical: Small formulas only

2. **Virtual Substitution**
   - Linear real arithmetic
   - Polynomial complexity for bounded degree
   - Widely used in practice

3. **Cooper's Algorithm**
   - Linear integer arithmetic
   - Complexity: O(n²ⁿ)
   - Used in SMT solvers

4. **Model-Based Elimination**
   - Generate instances
   - Check with SAT solver
   - Heuristic but often effective

### 5.5 Grounding

**Approach:** Replace quantified formulas with ground instances.

**Methods:**

1. **E-Matching**
   - Pattern matching on ground terms
   - Used in SMT solvers
   - Can miss relevant instances

2. **Instantiation**
   - Generate instances heuristically
   - Focus on relevant terms
   - Can be expensive

3. **Model-Based Instantiation**
   - Use model to guide instantiation
   - Iterate to refinement
   - More effective than E-matching

**Limitations:**
- Not complete (may not find proof)
- Can generate too many instances
- Requires good heuristics

## 7. Model Finding and Counterexample Generation

### 7.1 Finite Model Enumeration

#### 7.1.1 Brute Force Search

**Approach:**
- Enumerate all magmas of size n
- Check if E₁ holds and e₂ fails
- Return counterexample if found

**Complexity:**
- Number of magmas of size n: n^(n²)
- For n=2: 2^(4) = 16 magmas
- For n=3: 3^(9) = 19,683 magmas
- For n=4: 4^(16) ≈ 4.3 billion magmas

**Optimizations:**
- **Isomorphism elimination**: Factor out relabeling (n! reductions)
- **Symmetry breaking**: Fix some elements

### 8.1 Gröbner Basis Methods

#### 8.1.1 Foundations

**Polynomial Ideals:**
- Set of polynomials closed under addition and multiplication

#### 8.1.2 Buchberger's Algorithm

**Input:** Set of polynomials F

**Output:** Gröbner basis G

**Algorithm:**
```
1. G := F
2. P := {(f, g) | f, g ∈ G, f ≠ g}
3. While P ≠ ∅:
     a. Pick (f, g) from P
     b. Compute S-polynomial S(f, g)
     c. r := normal_form(S(f, g), G)
     d. If r ≠ 0:
          G := G ∪ {r}
          Update P
4. Return G
```

**Complexity:**

### 9.3 Rewriting Engine

#### 9.3.1 Matching

**Input:** Rule ℓ → r, term t

**Output:** Substitution σ such that σ(ℓ) = t

**Algorithm:**
- Similar to unification
- Only substitute variables in ℓ
- Directional matching

```python
def match(pattern, term):
    if pattern.is_var():
        return {pattern: term}
    if pattern.func != term.func:
        return None
    if len(pattern.args) != len(term.args):
        return None
    subst = {}
    for p, t in zip(pattern.args, term.args):
        s = match(p, t)
        if s is None:
            return None
        if conflict(subst, s):
            return None
        subst.update(s)
    return subst
```

#### 9.3.2 Rewriting Strategy

**Innermost:**
- Evaluate arguments first
- Similar to call-by-value
- Always terminates if system is terminating

**Outermost:**
- Evaluate at root first
- Similar to call-by-name
- Can be more efficient

**Parallel:**
- Rewrite all redexes simultaneously
- Maximum parallelism
- Requires dataflow analysis

```python
def rewrite(term, rules):
    # Try to match rules at root
    for rule in rules:
        subst = match(rule.lhs, term)
        if subst is not None:
            return apply(rule.rhs, subst)

    # No match at root, recurse
    if term.is_app():
        new_args = [rewrite(arg, rules) for arg in term.args]
        return Term(term.func, new_args)

    # Variable or constant
    return term
```

```python
class DiscriminationNet:
    def __init__(self):
        self.root = {}

    def insert(self, term, data):
        node = self.root
        for sym in term.to_path():
            node = node.setdefault(sym, {})
        node['data'] = data

    def query(self, pattern):
        return self._query(pattern, self.root)

    def _query(self, pattern, node):
        if 'data' in node:
            yield node['data']
        if pattern.is_var():
            for child in node.values():
                if child != 'data':
                    yield from self._query(pattern, child)
        else:
            sym = pattern.func
            if sym in node:
                yield from self._query(pattern.args[0], node[sym])

  | add x y ihx ihy =>
    simp [const_fold, eval]
    split
    . exact ihx
    . exact ihy
    . simp [ihx, ihy]
```

    _ = (a + b) * c + (a + b) * d := by rw [add_mul]
    _ = a*c + b*c + (a + b) * d := by rw [mul_add]
